Elicitar requisitos e histórias de usuário

Voltar para o diagrama

Descrição

Essa atividade está ligado ao levantamento de requisitos funcionais em formato de histórias de usuários (User Stories). Essa etapa é fundamental para a construção do Product Backlog, que depende das histórias de usuários para definir o que se deve desenvolver.

Responsáveis

Product Owner, e se for possível, um auxiliar ao Product Owner para que possa conduzir a escrita das histórias.

Entradas e saídas

Entrada

  • Necessidades e expectativas dos stakeholders

Saída

  • Como resultado, você consegue as Histórias de usuários, sendo uma coletânea de histórias que descrevem o que é esperado do produto a ser desenvolvido.

Tarefas

  • Realizar reuniões com os stakeholders para obtenção das necessidades e expectativas dos stakeholders
  • Registrar as histórias de usuário de acordo com o modelo proposto por Connextra (2021): "Como (papel), eu quero (algo) para (me beneficiar)".
  • Registrar requisitos não funcionais detectados

Ferramentas/técnicas auxiliares

Para executar essa atividade, é fundamental que o Product Owner defina uma forma das histórias ficarem organizadas. Portanto, pode definir um formato na qual elas podem ser escritar, como o modelo abaixo proposto por Connextra em 2021:

Como (papel), eu quero (algo) para (me beneficiar)

O exemplo abaixo demonstra a aplicação desse modelo

Como um gestor de atividades, eu quero poder cadastrar atividades e tarefas de um processo para eu poder organizar a sequência de atividades a serem realizadas em um projeto.

Revisar e corrigir histórias de usuários

Voltar para o diagrama

Descrição

Essa atividade refere-se à revisão e correção das histórias dos usuários, sendo fundamental tanto para garantir a qualidade e viabilidade dos requisitos quanto para a construção do Product Backlog.

Responsáveis

Product Owner, e se for possível, um auxiliar ao Product Owner, de preferência um desenvolvedor para verificar a viabilidade de atendimento das histórias de usuário.

Entradas e saídas

Entrada

  • Histórias de usuários

Saída

  • Como resultado, você consegue as histórias de usuários revisadas, sendo uma coletânea das histórias revisadas e analisadas.

Tarefas

  • Analisar histórias de usuário
  • Reunir e conversar com os stakeholders
  • Esclarecer dúvidas
  • Estudar viabilidade das histórias de usário

Ferramentas/técnicas auxiliares

Para maior clareza, as histórias podem estar definidas em um formato homogêneo, como o modelo Connextra desenvolvido em 2001 (BIK et al., 2017). Ao se utilizar um modelo, há maiores chances das histórias serem claras.

Como (papel), eu quero (algo) para (me beneficiar)

O exemplo abaixo demonstra a aplicação desse modelo

Como um gestor de atividades, eu quero poder cadastrar atividades e tarefas de um processo para eu poder organizar a sequência de atividades a serem realizadas em um projeto.

Materiais e referências

BIK, Niels; LUCASSEN, Garm; BRINKKEMPER, Sjaak. A reference method for user story requirements in agile systems development. In: 2017 IEEE 25th International Requirements Engineering Conference Workshops (REW). IEEE, 2017. p. 292-298.

Elaborar o product backlog

Voltar para o diagrama

Descrição

A elaboração do do produto backlog é uma atividade realizada pelo Product Owner (PO), com o objetivo de estabelecer uma lista a qual contem todos os requisitos necessários para o desenvolvimento do produto a partir das histórias de usuário as quais são colocadas em uma ordem de prioridade de acordo com a perspectiva do PO. A definição do Product Backlog deve sempre ocorrer antes do acontecimento de uma sprint, ou seja mesmo que o processo de desenvolvimento já tenha sido iniciado o Product Backlog pode ser alterado após a finalização da Sprint atual.

Note, que para garantir o bom desenvolvimento do produto, o Product Owner também deverá garantir de que todos os integrantes da equipe entendam as histórias de usuário e os requisitos delimitados no Product Backlog.

Responsáveis

Os responsáveis por esta atividade são o Product Owner juntamente com o seu ajudante se este existir. O PO deve definir a ordem de importância de cada item do Product Backlog.

Entradas e saídas

Entrada

  • Histórias de usuário revisadas

Saída

  • Product Backlog

Tarefas

  • Desdobrar as histórias de usuário em requisitos funcionais os quais irão compor o Product Backlog
  • Priorizar os items do product backlog

Ferramentas/técnicas auxiliares

As ferramentas utilizadas nesta etapa foram a plataforma de controle de versionamento de código Github em conjunto com a ferramenta Zenhub. Para facilitar o acesso a plataforma disponibilizada pelo Zenhub, foi-se utilizado um plugin para Google Chrome o qual permite o acesso do projeto a partir do próprio repositório do projeto.

4. Planejar sprint

Voltar para o diagrama

Descrição

Antes do início da Sprint, deve ocorrer o Sprint Planning, isto é, planejamento da Sprint. Artefatos como a Sprint Backlog e Definição de Pronto são definidas nessa etapa. Deve ser feita em uma reunião com no máximo 8 horas de duração (dependendo da duração da Sprint) (SCHWABER; SUTHERLAND, 2020).

A Sprint Backlog define um conjunto de atividades que deverão ser feita na Sprint após o planejamento. Esse conjunto de atividades são instanciadas das Histórias de Usuário selecionadas para ser concluída na Sprint. Observe que nem sempre pode ser possível concluir uma ou mais Histórias de Usuário em uma única Sprint, podendo ser encarregadas para a próxima.

A Definição de Pronto define um conjunto de critérios de conclusão para as atividades. Como cada atividade pode ser distinta entre si (por exemplo, atividades de codificação e atividades de elaboração de documentação), a Definição de Pronto pode especificar qual o critério de conclusão para cada tipo de atividade. Caso a Definição de Pronto elaborada em um planejamento anterior ainda esteja em conformidade e não necessite de alteração, o Team pode optar por pular esta etapa.

Responsáveis

Os responsáveis pelo planejamento da Sprint são todo o Team.

O Scrum Master é responsável por organizar e dar um fluxo contínuo à reunião, sempre levantando pontos de discussão importantes da reunião e permitindo que todos participem, cada um dando sua voz.

O P.O. é quem deve lembrar a equipe sobre as metas mais importantes a serem alcançadas para entrega do incremento, utilizando como base o Product Backlog já desenvolvido. Ele é quem deve dar a visão geral e também sugerir qual o objetivo principal a ser alcançado pela Sprint atual.

Todo o Team deve discutir e selecionar quais as Histórias de Usuário que serão encarregadas para a Sprint atual. É importante que todos opinem em cima da seleção, tendo sempre em mente o que se deseja entregar para aquela Sprint.

O Time de Desenvolvimento será responsável por instanciar as atividades com base nas Histórias de Usuário selecionadas. Essas atividades devem ser estimadas seguindo alguma técnica: sugere-se o uso do Planning Poker, técnica na qual os desenvolvedores estimam os pontos (ou tamanho) das atividades. Essa técnica permite que todos os desenvolvedores votem pontos para uma atividade em pauta e discutem sobre o porquê deram aquele ponto, de forma a se chegar uma conclusão.

Mais detalhes de como realizar o Planning Poker pode ser encontrado nesse post feito por Niladri Mahapatra (2017). Observe que apenas as atividades devem ser estimadas, visto que as Histórias de Usuário são apenas referências de metas que se deseja alcançar.

É possível que a soma dos pontos das atividades definidas naquela Sprint exceda a capacidade de trabalho da equipe. Caso isso ocorra, sugere-se adiar algumas das atividades (ou até mesmo Histórias de Usuário) para uma futura Sprint. É sempre importante ter em mente qual a capacidade total de trabalho da equipe.

Entradas e saídas

Entrada

  • Product Backlog

Saída

  • Sprint Backlog
  • Definição de Pronto (Definition of Done, DoD)

Tarefas

  • Elaborar, caso não se tenha ou não esteja em conformidade, uma Definição de Pronto
  • Definir o objetivo da Sprint atual
  • Selecionar as Histórias de Usuário relevantes para o objetivo da Sprint
  • Instanciar atividades com base nas Histórias de Usuário
  • Estimar os pontos das atividades instanciadas

Ferramentas/técnicas auxiliares

  • Planning Poker Online para estimar os pontos das atividades
  • Zenhub para Github para construir o Sprint Backlog com um conjunto de Histórias de Usuário e atividades

Materiais e referências

5. Executar atividades da Sprint

Voltar para o diagrama

Descrição

A execução das atividades da Sprint se referem às atividades do Sprint Backlog definido anteriormente. Essas atividades serão executadas por todo o Time de Desenvolvimento no período estipulado para a Sprint, sem mais ou menos atividades: a Sprint Backlog e a Definição de Pronto não pode ser alterada durante essa execução.

O incremento a ser desenvolvido durante essa etapa pode ser constituído por diversos artefatos, como documentação, codificação e elaboração de casos de testes. Os métodos e ferramentas já devem ter sido definidos na atividade de (TODO LINK) "Definir ferramentas, métodos e duração da Sprint", utilizando padrões como a UML e técnicas como o Test-Driven Development.

Para se ter um desenvolvimento fluido, é importante se ter uma paralelização do desenvolvimento, com cada membro do Time de Desenvolvimento se encarregando por um conjunto de atividades de uma parte do incremento. Os membros também devem ter consciência da Definição de Pronto, e antes de encarregar uma atividade como concluída, precisam atingir o requisito da Definição de Pronto (definido na atividade "4. Planejar sprint").

Para manter o progresso das atividades, o Time de Desenvolvimento precisa registrar elas em um quadro visual, com cada atividade encarregada para algum membro. Pode ser utilizado ferramentas como o ZenHub (2021), Trello (2021), Jira (2021) ou até mesmo um quadro Kanban físico. Sugere-se o uso do ZenHub ou Trello, ambos são gratuitos.

O Time de Desenvolvimento deve realizar a reunião diária para esclarecer qualquer problema e manter uma fluidez com o desenvolvimento.

Para acompanhar o progresso do desenvolvimento, o Time de Desenvolvimento pode utilizar o gráfico de Burndown, que mede uma linha ideal de progresso de conclusão das atividades durante o tempo da Sprint. Se a equipe estiver acima da linha, está atrasada, enquanto se estiver abaixo da linha, está adiantada. Esse gráfico é importante para se ter noção de como o Time de Desenvolvimento está com o desenvolvimento. O ZenHub já possui um Burndown integrado, e o Trello aceita como uma extensão ao projeto.

Ao fim da execução das atividades, se espera ter um incremento pronto. É possível que nem todas as atividades possam ter sido realizadas com sucesso, as quais devem ser encarregadas para uma próxima Sprint. Caso isso ocorra, é importante discutir futuramente na retrospectiva da Sprint os problemas enfrentados pelos membros do Time de Desenvolvimento.

Responsáveis

Apenas o Time de Desenvolvimento trabalham nessa atividade. Caso surja dúvidas com alguma História de Usuário, essas dúvidas devem ser registradas futuramente para ser discutida com o P.O. (mas sempre lembrando que a Sprint Backlog atual não pode ser alterada). Observe que o Scrum Master também pode atuar como desenvolvedor e fazer parte do Time de Desenvolvimento.

Entradas e saídas

Entrada

  • Sprint Backlog
  • Definição de Pronto

Saída

  • Um incremento desenvolvido, constituído por:
    • Documentação
    • Código da funcionalidade
    • Casos de testes

Tarefas

  • Executar as atividades planejadas no Sprint Backlog
  • Elaborar a documentação do incremento
  • Codificar as funcionalidades principais
  • Escrever e codificar os casos de testes

Ferramentas/técnicas auxiliares

  • Zenhub para Github para manter o progresso das atividades.
  • Trello para manter o progresso das atividades.
  • Jira para manter o progresso das atividades.

Materiais e referências

  • BUTTLER, Matt. How the ZenHub Team Uses ZenHub Boards on GitHub. 2014. Disponível em: https://blog.zenhub.com/how-the-zenhub-team-uses-zenhub-boards-on-github/
  • CAMARGO, Robson. Burndown: conheça o gráfico que mede produtividade. 2020. Disponível em: https://robsoncamargo.com.br/blog/Burndown/

Realizar reunião diária

Voltar para o diagrama

Descrição

É uma reunião rápida com duração de até 15 minutos realizada todos os dias. É importante que a reunião seja realizada no início do expediente.

Responsáveis

Os responsáveis pelas reuniões são todos da equipe de desenvolvedores. O Product Owner e o Scrum Master podem participar das reuniões como ouvintes.

Entradas e saídas

Entrada

  • Sprint Backlog
  • Feedback do dia anterior da equipe que fica registrado no Discord

Saída

  • Feedback da equipe no presente dia

Tarefas

  • Levantar o que fei feito desde a última reunião
  • Levantar o que será feito no dia em questão
  • Discutir quais os impedimentos para a realização do trabalho

Ferramentas/técnicas auxiliares

  • Discord

Revisar sprint (sprint review meeting)

Voltar para o diagrama

Descrição

A Sprint review é o momento em que a equipe e as partes interessadas conversam sobre o andamento final da sprint, validando as entregas da equipe, verificando se o projeto está de acordo com o estabelecido inicialmente e também o mais importante, que é a coleta do feedback do que foi construido pela equipe.

Responsáveis

O Scrum Team e os stakeholders. Caso não seja possível a presença de todos stakeholders, apenas a presença do PO para representatividade para os stakeholders

Entradas e saídas

Entrada

  • Incremento desenvolvido

Saída

  • Feedback sobre o produto das partes interessadas

Tarefas

  • Apresentação das entregas feitas pela equipe
  • Feedback do Product Owner e/ou Stakeholders
  • Validação de conclusão da entrega de acordo com os requisitos necessários para a conclusão da sprint
  • Verificar se a entrega está de acordo com a definição de pronto da sprint

Ferramentas/técnicas auxiliares

  • Github
  • Zenhub para Github

Realizar retrospectiva da sprint

Voltar para o diagrama

Descrição

Esta atividade consiste numa reunião realizada ao final da sprint a fim de gerar ações para melhorar o processo de trabalho, melhorando de forma contínua e incremental, ou seja, melhore de pouquinho em pouquinho, de forma regular e consistente.

Responsáveis

Todo o Team

Entradas e saídas

Entrada

  • Feedback do incremento
  • Sprint Backlog
  • Product Backlog

Saída

  • Feedback e avaliação da Sprint

Tarefas

  • Reunir com a equipe de desenvolvimento
  • Cada integrante do time menciona um ponto positivo, negativo e o que melhorar e o explica para a equipe
  • Anotar os pontos positivos e negativos, para aplicar futuramente na próxima Sprint Planning

Ferramentas/técnicas auxiliares

  • Github
  • Zenhub para Github

Verificar itens restantes no Product Backlog

Voltar para o diagrama

Descrição

Essa atividade de verificação serve apenas para verificar se ainda restam itens que devem ser concluídos no Product Backlog. Caso exista, é importante ter a continuidade do projeto, e caso contrário, o projeto estará finalizado.

Responsáveis

Basta o PO verificar se ainda restam requisitos a serem concluídos.

Entradas e saídas

Entrada

  • Product Backlog

Saída

  • Avaliação do estado do Product Backlog

Tarefas

  • Avaliar o estado atual do Product Backlog
  • Verificar quantidade de itens já atendidos

Ferramentas/técnicas auxiliares

  • Github
  • Zenhub para Github

Saída
